# 软光栅渲染器项目实现



---

## 一、项目概述

**项目名称**：CPU Soft Rasterizer（C++17 软件光栅渲染器）  
**技术栈**：C++17、Win32 GDI、手写线代与渲染管线（无 GPU / 无第三方渲染库）  
**定位**：从零实现可编程管线风格的离线/实时软光栅，覆盖 MVP 变换、裁剪、光栅化、Z-Buffer、光照与常见后处理拓展。

**一句话简介**：独立实现一套完整的 CPU 实时软光栅管线，并在场景级完成点光源 Shadow Map、半透明双 Pass、Bloom、色彩分级等进阶效果，系统排查并修复矩阵语义、背面剔除、光照一致性等关键问题。

---

## 二、系统架构与核心能力

### 模块划分

| 层次 | 模块 | 职责 |
|------|------|------|
| 数学层 | `Vec2/3/4`、`Mat4` | 齐次坐标、行列式布局一致的变换、LookAt / Perspective |
| 场景层 | `Mesh`、`SceneObject`、`Camera` | 几何生成（立方体/地板/球/墙面）、材质属性、轨道相机 |
| 光栅核心 | `Rasterizer`、`Framebuffer`、`Interp` | MVP → CVV 裁剪 → 三角形填充 → 深度测试 → 着色 |
| 光照与贴图 | `Lighting`、`Texture`、`MaterialLibrary` | Blinn-Phong（方向光/点光）、反照率 + 法线贴图 |
| 阴影 | `ShadowMap` | 光源视角深度缓冲、径向距离比较、2×2 PCF |
| 后处理 | `BloomPass`、`ColorGrade` | 泛光提取/模糊/合成；亮度/饱和度/对比度 |
| 交互 | Win32 主循环 | 相机轨道/平移、模型旋转、快捷键、GDI Overlay UI |

### 已实现能力清单

1. **几何管线**：Model–View–Projection、透视除法、齐次裁剪（Near + 六面 CVV）
2. **光栅化**：重心坐标插值、透视校正（`invW`）、线框/实体模式切换
3. **深度**：Early / Late Z-Buffer（写深度 / 透明只测不写）
4. **抗锯齿**：每像素 4 子采样 MSAA（颜色 + 阴影图同步）
5. **着色**：Blinn-Phong；点光源衰减；法线贴图（TBN）
6. **阴影**：点光源 Shadow Map；径向距离比较；偏置；PCF；半透明投影强度衰减
7. **透明**：不透明 → 半透明两 Pass；SrcOver 混合
8. **后处理**：Emissive Bloom + 近白区域边缘 Bloom；色彩分级滑条实时调节
9. **场景表达**：墙体布尔镂空落地窗、灰色材质统一、白色背景环境光语义

## 三、关键难点与解决方案（SOP）

### 1. LookAt 矩阵行列式布局与变换约定不一致 → 灰屏 / 几何被裁尽

**现象**：相机摆到等轴测位置后画面只剩背景，几何全部消失。  
**根因**：`lookAt` 按列向量约定存储旋转，但 `transformPoint` 按**行向量点积**读矩阵，导致 `clip.w` 符号错误，三角形在齐次近裁剪阶段被全部丢弃。  
**解决**：统一矩阵约定——LookAt 改为与 row-dot 变换一致的行主序布局；相机以「看目标点 + 球坐标轨道」方式初始化。  
**收获**：图形程序中「数学形式」必须与「内存访问约定」绑定验证，矩阵问题常表现为「看不见」而非「算错一点」。

### 2. Shadow Map 误用 clip 空间深度

**现象**：球体背面/受光面阴影投印不符合直觉。  
**根因**：点光源不能像平行光那样只用单一方向的深度比较；用 `clip.w` 等非径向量比较，与灯光几何不一致。  
**解决**：Shadow Pass 与 Color Pass 统一存/比较**世界空间到点光源的径向距离**，并加 bias；仅对 `N·L > 0` 的面片做阴影测试。  
**收获**：阴影算法要贴合灯光模型；点光源 Shadow Map 的本质是距离场比较，而非方向光 depth map 的简单复用。

### 3. 墙面「建了却看不见」——绕序与背面剔除

**现象**：只能看到一面墙，带窗墙面缺失。  
**根因**：PosX 墙网格顶点绕序与约定法线方向相反，光栅器屏幕空间面积判定（背面剔除）将其剔除。  
**解决**：与立方体 +X 面统一绕序；法线朝向室内 (+X)。  
**收获**：几何法线、索引绕序、剔除符号三者必须一致，调场景几何时优先做「绕序 sanity check」。

### 4. 关阴影后地面「固有色变浅」——功能开关误切光照模型

**现象**：按 L 关闭投影后，地面明显变亮，与墙面预期不符。  
**根因**：阴影开关同时切换了 PointLight → DirectionalLight；平行光俯射使水平地面近乎全漫反射，再加上地面 albedo 本就更亮。  
**解决**：L **只开关 Shadow Map**；着色始终点光；关阴影时 `shadowFactor=1`；统一地面/墙面 albedo。  
**收获**：产品级开关应「正交」：关掉 A 不应偷偷改变 B 的光照约定。

### 5. 半透明物体阴影应更「淡」

**现象**：透明立方体仍投下与不透明相同的实心阴影，物理直觉差。  
**解决**：Shadow Map 增加 `castShadowStrength` 通道；半透明写入 strength=α；PCF 合成时按遮挡强度缩放阴影深度。  
**收获**：软阴影不只是「有/无」，遮挡透明度也可建模为强度调制。

### 6. Bloom 与窗洞「洞口光」一体化

**需求**：窗洞看向白背景时应有边缘泛光；物体挡住窗洞交界仍保留光晕。  
**解决**：光源球 Emissive 写入 bloom 缓冲；MSAA Resolve 后对近白像素做 8 邻域轮廓提取再模糊合成——语义上把「无限远高亮背景」当作 bloom 源。  
**收获**：后处理可以不依赖 HDR 浮点缓冲，在 LDR 软光栅上用「语义阈值/边缘」近似高亮溢出。

### 7. 全屏 BitBlt 与 Win32 子控件 Z 序冲突

**现象**：左下角 Trackbar 场景加载后消失，甚至初始帧也看不见。  
**解决**：舍弃子控件，改为每帧 `BitBlt` 后 **GDI Overlay 自绘滑条**，自管 hit-test 与拖拽。  
**收获**：实时全屏 blit 应用里，UI 与呈现管线耦合时，自绘制 overlay 往往比「真控件置顶」更可控。

## 四、技术收获

1. **管线贯通**：亲手走完顶点变换 → 裁剪 → 光栅 → 片段着色 → 后处理，对 GPU 黑盒步骤建立可调试的心智模型。  
2. **数值与约定**：齐次坐标、透视校正插值、矩阵布局、绕序/剔除符号——图形 bug 多为约定一致性问题。  
3. **算法选型能力**：在 CPU 约束下选择可实现的抗锯齿（MSAA）、软阴影（PCF）、Bloom（可分离盒模糊）与半透明（双 Pass）折中方案。  
4. **调试方法论**：灰屏先查 `clip.w` 与裁剪；阴影错先核对灯空间几何；表面「颜色不对」先分清 albedo、光照模型、后处理是否被开关误伤。  
5. **工程化意识**：功能正交（阴影开关 ≠ 换灯）、材质与几何分离、场景对象统一抽象（cast/receive/alpha/strength）。

## 六、面试口述结构（约 2–3 分钟）

1. **目标**：理解渲染管线，而不是调引擎参数。  
2. **核心链路**：CPU 上复刻光栅管线 + 点光阴影场景。  
3. **一个深挖点**（任选）：  
   - LookAt/行主序导致的灰屏；或  
   - 点光径向 Shadow Map vs 方向光 depth；或  
   - 半透明双 Pass + 阴影强度。  
4. **结果**：可交互实时场景，支持阴影/透明/Bloom/调色开关，并有可复述的排错闭环。
